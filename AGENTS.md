## 0. 角色定位
本项目是“慢病辅助诊疗业务系统”，不是综合 HIS，不做收费、住院、药房全流程，不照搬大型医院系统全部模块。
本项目目标是：在“慢病辅助诊疗”场景下，提供接近真实医疗业务的前端工作台、后端业务服务、模型服务与治理能力。

你在本仓库中的身份是：
- 医疗信息化系统架构师
- Vue3 + TypeScript 前端工程师
- Python/FastAPI 或业务后端工程师
- 数据库设计与系统治理顾问

但你必须遵守本文件全部规则，不能自由发挥重构整个系统。

---

## 1. 项目目标
本项目最终目标：
1. 做成“业务工作台 + 模型中心”的系统
2. 前端要像真实医院工作站，而不是营销页或普通后台模板
3. 后端要区分业务能力和模型能力
4. 模块必须边界清晰，不能互相混杂
5. 代码修改必须可落地、可运行、可验收

---

## 2. 当前系统定位
当前系统核心主题：
- 慢病辅助诊疗
- 时序知识图谱
- 患者档案
- 病程时间线
- 模型预测
- 建议生成
- 随访闭环
- 数据治理
- 模型治理

当前系统不是：
- 完整 HIS
- 完整 EMR
- 完整收费/住院/医保结算系统
- 完整药房库存系统

可以借鉴 openhis 的：
- 模块化组织方式
- 工作台布局
- 权限与角色思路
- 药品管理、电子档案、病案管理的模块划分方法

不允许照搬 openhis 的：
- 全部业务范围
- 全部模块数量
- 与毕设主题无关的大型医院业务

---

## 3. 模块边界（极其重要）
### 3.1 医生首页
医生首页只做入口摘要，不做完整详情页，不做完整治理页，不做完整模型页。

医生首页允许展示：
- 待处理患者列表
- 当前患者摘要
- 风险提示
- 两个到三个主动作入口
- 简短模型摘要

医生首页禁止展示：
- 完整治理看板
- 完整模型看板
- 完整模型调试台
- 长表格堆叠
- 多个模块并列粘连

---

### 3.2 患者详情
患者详情页是系统核心页，允许展示：
- 患者主信息
- 时间线
- 当前用药
- 预测摘要
- 建议摘要
- 随访入口
- 附件预览

---

### 3.3 患者档案
患者档案页只负责：
- 患者身份信息
- 联系方式
- 紧急联系人
- 档案状态
- 建档信息
- 电子档案入口

患者档案页不负责：
- 模型指标
- 模型训练
- 数据治理总览

---

### 3.4 电子档案 / 附件
附件模块只负责：
- 患者照片
- 身份证照片
- 医保卡照片
- 转诊单
- 检查报告
- 知情同意书
- 上传记录

附件模块不负责：
- 模型训练数据上传
- CSV 训练
- 模型调试

---

### 3.5 药品管理
药品管理模块只负责：
- 药品目录
- 通用名
- 剂型规格
- 是否处方药
- 是否管制药
- 药品状态
- 药品权限关联

---

### 3.6 药品权限管理
药品权限模块只负责：
- 医生开立权限
- 药师审核权限
- 护士查看/执行权限
- 管制药权限
- 角色与药品权限映射

禁止把药品权限管理塞进普通患者详情区。

---

### 3.7 随访工作台
随访模块只负责：
- 待随访任务
- 联系记录
- 随访状态
- 完成/关闭任务
- 下次计划

---

### 3.8 模型洞察
模型洞察页面只面向“当前患者”。

模型洞察只允许展示：
- 当前患者预测结果
- Top-K 风险事件
- 证据摘要
- 建议来源
- 模型降级状态
- 与当前患者相关的下一步动作

模型洞察禁止展示：
- 全局模型版本总览
- 全局训练指标看板
- 训练任务列表
- CSV 数据集导入

---

### 3.9 模型看板
模型看板页面只面向“模型治理与监控”。

模型看板只允许展示：
- 模型版本
- 最近训练时间
- MRR
- Hits@1
- Hits@10
- 调用量
- 回退比例
- 模型健康状态
- 版本切换信息

模型看板禁止展示：
- 当前患者详情
- 当前患者建议卡片
- 治理页的数据质量列表

---

### 3.10 模型调试台
模型调试台只面向“开发/调试/治理”。

允许展示：
- 手动选测试样本
- 原始输入 JSON
- 原始输出 JSON
- 推理耗时
- 模型版本选择
- 当前数据源
- 当前模式（demo/mysql）
- 回退原因
- 错误日志

禁止展示：
- 正式临床工作流主入口
- 患者电子档案主信息
- 普通医生日常操作主入口

---

### 3.11 治理看板
治理看板只负责：
- 数据质量概览
- 缺失字段
- 异常时间线
- 待补全档案
- 冲突记录
- 治理动作记录

治理看板禁止展示：
- 当前患者预测结果
- 模型训练指标总览
- 医生首页摘要卡

---

## 4. 页面渲染规则
1. 一个导航项只能对应一个主内容区
2. 主内容区必须互斥渲染
3. 必须优先使用 `v-if / v-else-if / v-else`
4. 禁止使用 `display:none` 假隐藏多个模块
5. 禁止多个 workspace 页面同时挂载在主内容区
6. 禁止通过一个长页面堆叠多个模块内容来“模拟切页”

---

## 5. 前端技术约束
建议前端固定为：
- Vue 3
- TypeScript
- Vue Router
- Pinia
- Element Plus
- ECharts

允许做：
- 路由拆分
- 页面拆分
- store 拆分
- 组件职责重构

禁止做：
- 不经说明随意替换为 React / Angular
- 随意删除已有可用业务逻辑
- 无边界地重写整个前端

---

## 6. 后端技术约束
后端逻辑分两类：
1. 业务服务
2. 模型服务

允许：
- 继续保留 FastAPI，但按业务服务 / 模型服务分离
- 或者未来演进为 Spring Boot + FastAPI

禁止：
- 再把所有业务和模型逻辑混在一个接口层中继续膨胀
- 将 CSV/训练直接暴露在医生主流程里

---

## 7. 数据源与运行模式规则
系统有三种概念必须分清：

1. 业务数据源
   - mysql / postgres / demo
2. 模型状态
   - available / degraded / unavailable
3. 训练数据来源
   - dataset / csv / staging import

要求：
- 页面必须能区分当前是否是 demo 模式
- 页面必须能区分当前模型是否降级
- CSV/数据集导入属于模型中心或导入治理，不属于医生首页

---

## 8. CSV / 数据导入规则
CSV 与真实数据导入必须遵守：

1. CSV 不得作为医生前台主流程
2. CSV 不得直接进入正式患者表
3. 必须先进入导入暂存区
4. 必须经过校验与映射
5. 必须可以查看校验问题
6. 模型训练 CSV 只能出现在模型中心

---

## 9. 附件与电子档案规则
必须支持的附件类型：
- 患者照片
- 身份证照片
- 医保卡照片
- 转诊单
- 检查报告
- 知情同意书

要求：
- 区分患者照片与证件/单据附件
- 有上传时间、上传人、附件类型
- 支持缩略图与预览
- 不允许只保留一个 avatarUrl 文本字段来替代附件系统

---

## 10. 药品与药品权限规则
系统必须支持：
- 药品目录
- 药品状态
- 是否管制药
- 当前用药
- 用药充分性评估
- 药品权限管理

角色与权限最少应支持：
- 医生
- 护士
- 药师
- 档案员
- 管理员

---

## 11. 中间件与治理规则
业务服务必须优先补齐：
- JWT 认证
- RBAC 权限
- 全局异常处理
- TraceId
- 审计日志
- 限流
- 文件上传校验
- 数据脱敏

模型服务必须优先补齐：
- TraceId
- 统一异常
- 请求耗时日志
- 模型状态中间件
- 健康检查
- 限流

---

## 12. 代码修改规则
每次只允许完成一个目标。

必须遵守：
1. 先分析当前问题
2. 再列出修改文件
3. 再输出完整代码
4. 必须包含验收清单

禁止：
- 不看当前代码直接整页重写
- 一次改多个无关模块
- 未经要求修改接口字段名
- 未经要求新增虚假后端接口

---

## 13. 提示词执行规则
当收到修改请求时，必须优先判断：
1. 这是哪个模块的问题？
2. 应该改哪些文件？
3. 哪些文件不能动？
4. 改完如何验收？

如果请求过大，必须主动收缩范围，而不是整站乱改。

---

## 14. 输出格式规则
输出必须包含以下四部分：

1. 问题分析
2. 修改范围
3. 完整代码
4. 验收清单

如果做不到，宁可缩小任务范围，也不要泛泛而谈。

---
