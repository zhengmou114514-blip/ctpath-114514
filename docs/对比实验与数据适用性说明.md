# 对比实验与数据适用性说明

## 1. 为什么这样设计对比实验

任务书要求当前系统不仅要完成 Web 可视化和病情推演，还需要通过实验说明时序知识图谱方法优于传统静态知识图谱方法。

结合现有仓库代码，最合适、也最容易复现的静态 KG 对比模型是：

1. `TransE`
2. `RotatE`

选择理由：

1. 它们都是经典静态知识图谱嵌入模型，符合任务书里“传统静态 KG 模型”的对比要求。
2. 仓库中已经有可运行实现，避免再引入新的代码库，方便保证实验闭环。
3. `TransE` 代表平移式关系建模，`RotatE` 代表旋转式关系建模，覆盖两类典型静态 KG 方法。

## 2. 新增的对比实验脚本

新增脚本：

- `models/run_static_baselines.py`

它做了几件事：

1. 读取 `data/<dataset>/train|valid|test`
2. 为静态 KG 基线补 reciprocal triples
3. 统一按“tail + reciprocal head”双向评估
4. 输出 `MRR / Hits@1 / Hits@3 / Hits@10`
5. 自动生成 `comparison.json` 和 `comparison.md`

## 3. 运行方式

在项目根目录执行：

```powershell
conda activate ctpath
cd E:\CTpath-master

python models\run_static_baselines.py `
  --dataset CHRONIC `
  --models TransE,RotatE `
  --embedding_dim 200 `
  --learning_rate 0.001 `
  --margin 1.0 `
  --num_neg 4 `
  --max_epochs 50 `
  --batch_size 1024 `
  --device cpu
```

结果会保存在：

- `models/baseline_results/CHRONIC_<timestamp>/comparison.json`
- `models/baseline_results/CHRONIC_<timestamp>/comparison.md`

## 4. 当前 CHRONIC 数据是否符合要求

### 4.1 满足的部分

当前 `CHRONIC` 已经满足模型格式要求：

1. 每条样本都是四元组 `(s, r, o, t)`
2. 已经按时间切分为 `train / valid / test`
3. 可以正常走 `process_icews.py -> path_data.py -> init_rw.py -> gen_merw -> learner.py`

所以从“模型能否训练”和“系统能否联调”来看，它是合格的。

### 4.2 不足的部分

当前 `CHRONIC` 更像是把一个慢病宽表按字段拆成很多离散关系，而不是原生临床时序事件流。

已有统计表现出几个明显特征：

1. 关系种类 `24` 个，但大量关系频次完全一样
2. 对象词表只有 `37` 个，偏小
3. 患者 `500` 左右，时间点 `1022`
4. 单个患者事件数偏高，说明“一行记录拆成多条特征事件”的痕迹明显

这意味着：

1. 它适合做系统演示、模型迁移验证、毕设功能展示
2. 它不适合单独支撑“真实临床有效性”这种强结论

## 5. 新增的数据体检脚本

新增脚本：

- `preprocess/audit_tkg_dataset.py`

执行：

```powershell
python preprocess\audit_tkg_dataset.py --dataset CHRONIC
```

它会检查：

1. 四元组格式是否正确
2. 时间切分是否存在泄漏
3. 是否呈现宽表特征扩展模式
4. 是否适合直接作为临床时序图谱

并生成报告：

- `docs/CHRONIC_data_audit.md`

## 6. 任务书视角下的结论

最稳的写法是：

1. `ICEWS14` 用来说明模型本身可复现
2. `CHRONIC` 用来说明模型已迁移到慢病辅助诊疗场景
3. `TransE / RotatE` 用作静态 KG 对比基线
4. Web 系统部分展示患者管理、病情时间线、T+1 预测和辅助建议

这样就能同时满足：

1. 算法实验
2. 对比实验
3. 医疗场景迁移
4. 系统实现

## 7. 如果当前慢病数据不够好，推荐怎么补

### 优先级 1：保留当前 CHRONIC，定位为“演示与迁移验证数据”

优点：

1. 已经能跑通
2. 已经接入前后端和 MySQL
3. 最适合保证毕设进度

### 优先级 2：增加真实纵向医疗数据

更适合 TKG 的数据应满足：

1. 同一患者有多次就诊或检查记录
2. 每次记录有明确时间
3. 能抽成 `(患者, 关系, 事件对象, 时间)` 四元组

推荐方向：

1. `MIMIC-IV`：真实 ICU/EHR 时序数据，最适合时序图谱，但获取门槛较高
2. `eICU-CRD`：也是纵向 ICU 数据，适合补真实临床时序
3. `DiaKG`：适合作为医疗知识图谱底座，但单独拿来做时序推理还不够，需要叠加患者时间事件

### 优先级 3：如果只能拿到表格数据

像糖尿病、CKD、心衰这类表格数据：

1. 可以用于系统页面展示
2. 可以作为传统分类基线
3. 不建议直接当成时序 KG 主实验数据
