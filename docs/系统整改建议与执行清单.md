# 系统整改建议与执行清单

## 1. 文档目的

本文档用于汇总当前项目代码在真实医院场景下的主要不足，并形成可持续回填的整改清单。后续每完成一项修改，都可以直接在本文件中补充状态、说明和验证结果，避免分析与实现脱节。

## 2. 当前代码现状判断

结合当前目录下的前后端代码，系统已经具备以下基础能力：

- 已完成患者建档、患者详情、事件录入、预测展示、建议生成等主流程。
- 已补充更贴近医院场景的档案字段，如头像、联系电话、紧急联系人。
- 已新增联系记录模块，并打通到患者详情页与随访工作台。
- 已将随访场景拆分为桌面端工作台方案和手机端首页方案，两端界面定位较为合理。

目前最突出的不足，不再是“有没有页面”，而是“系统底座是否足够真实、可维护、可继续扩展”。因此后续工作建议优先放在结构整理、安全性、流程联动和真实业务字段完善上。

## 3. 重点整改建议

### 3.1 高优先级

1. 认证与权限机制需要升级  
当前医生账号仍采用明文密码存储和直接比对，真实医院系统中至少应改为密码哈希、登录令牌过期控制、角色权限区分，并限制不同角色的可见菜单与可用接口。

2. 后端存在重复定义的核心函数  
`app/store.py` 中 `get_followup_worklist`、`get_flow_board` 存在重复实现，后定义会覆盖前定义，容易导致新增字段或逻辑失效。这类问题应优先清理，否则后续每次扩展都可能出现“代码写了但没生效”的情况。

3. 异常兜底方式过宽  
当前多处直接使用 `except Exception` 兼容旧表结构。演示阶段可接受，但真实系统应改为显式迁移、版本检测、明确日志和有限兜底，避免把真实错误吞掉。

4. 头像仍是文本链接方案  
目前头像字段依赖前端录入 URL，缺少真实上传流程。后续应补充上传接口、文件大小校验、默认头像和静态资源访问路径。

### 3.2 中优先级

1. 联系记录应与随访任务联动  
“未接通”“约定复联”“建议门诊复查”等联系结果，不应只停留在记录层，而应推动随访任务状态变化或新增待办。

2. 患者档案字段仍偏演示化  
建议继续补充身份证脱敏号、医保类型、责任医生、责任护士、建档科室、过敏史、家族史、照护依赖等字段，以支撑真实门诊和慢病管理场景。

3. 枚举与字典体系需要收口  
疾病、风险、关系类型、任务状态等字段仍有较多自由文本，后续应统一为后端枚举和前端字典映射，降低脏数据风险。

4. 缺少完整审计链  
系统需要逐步补充“谁在什么时候修改了什么”的审计记录，特别是患者档案、门诊任务和联系记录。

5. 会话令牌仍是进程内临时状态  
当前 token 主要保存在后端进程内存中，未设置明确过期时间，也没有持久化或刷新机制。真实医院系统中，这会带来重启失效、会话不可审计、无法统一登出等问题，后续应升级为带过期时间的签名令牌或集中式会话存储。

6. 部分接口与部署配置仍偏演示环境  
当前 CORS 主要面向本地开发地址，且部分接口能力仍存在待完善位置，例如建议链路中仍有“从数据库加载预测结果”的 TODO。后续若面向真实部署，需要把环境配置、来源校验和未闭环接口继续补齐。

### 3.3 端侧体验建议

1. 桌面端继续作为工作台  
桌面端适合保留多面板、多患者、多状态协同的工作方式，不建议为了统一而牺牲信息密度。

2. 手机端继续作为任务首页  
手机端应坚持任务流设计，重点放在“当前患者、快速记录、自动切换下一位患者、待复联提醒”等单任务高频操作上。

3. 两端保持一套数据链路、两套界面方案  
当前这个方向是正确的，后续不要回到“同一页面纯响应式缩放”的思路。

## 4. 建议执行顺序

建议按下面顺序逐步实施：

1. 清理 `app/store.py` 重复实现，恢复单一可信逻辑。
2. 收敛宽泛异常处理，补充结构迁移说明。
3. 升级登录认证与角色权限控制。
4. 让联系记录驱动随访任务联动。
5. 增加真实头像上传与更完整档案字段。
6. 继续增强手机端随访流程，如自动切换下一患者、复联提醒。

## 6. 本轮修改说明

本轮已先完成两项高优先级修正：

1. 已收敛 `app/store.py` 中对外生效的随访工作台函数定义  
当前仅保留一套实际生效的 `get_followup_worklist` 与 `get_flow_board`，避免后定义覆盖前定义；同时恢复了门诊任务的最近操作人、最近操作时间返回字段，便于桌面端工作台继续展示。

2. 已完成密码哈希升级并兼容旧数据  
新增 `app/security.py`，使用 `pbkdf2_sha256` 进行密码哈希；新注册账号默认写入哈希值；旧库中若仍为明文密码，可继续登录，并会在登录成功后自动升级为哈希存储。演示模式与 MySQL 模式均已接入，初始化 SQL 也已补充哈希升级语句。

3. 已收敛数据库结构兼容的异常兜底方式  
`app/store.py` 中患者扩展字段、门诊任务附加字段、联系记录等兼容逻辑，已从“直接捕获所有异常后回退”改为“先检测表字段是否存在，再选择对应 SQL”。这样能够减少误吞真实数据库错误，提高问题定位效率。

本轮验证结果：

- 前端 `npm run build` 通过。
- 后端 `py_compile` 通过。

下一步建议优先处理会话令牌与权限控制，把当前仅适合演示环境的登录态管理升级为更接近真实系统的方案。
