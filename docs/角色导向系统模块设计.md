# 角色导向系统模块设计

## 1. 设计目标

后续系统改造的重点从“功能堆叠”切换为“面向真实使用人员的系统设计与实现”。  
在慢性病辅助诊疗场景中，系统不应只服务单一的“模型演示者”，而应区分不同岗位的工作目标、信息密度和操作习惯。

本项目后续设计遵循以下原则：

- 预测与建议分层：`CTpath` 负责四元组时序预测，`DeepSeek` 负责建议解释与表达。
- 角色驱动界面：医生、护士/慢病管理师、档案员看到的模块不同。
- 业务模块先于展示模块：优先建设工作台、随访队列、流程状态和档案录入，不再优先扩展指标展示页。
- 页面结构向真实医院系统靠拢：参考开源医院信息系统的成熟布局，而非继续维持单页演示型界面。

## 2. 角色划分

### 2.1 医生

医生关注的是“当前这个患者值不值得优先处理、为什么、下一步该怎么做”。

医生端核心模块：

- 医生工作台
- 患者详情页
- N+1 预测页
- 证据四元组页
- 风险建议页

医生端重点信息：

- 患者横幅信息：姓名、编号、主要疾病、阶段、最近就诊
- 关键风险摘要
- 四元组证据
- CTpath Top-K 预测
- DeepSeek 生成的解释性建议

医生端不应承担的内容：

- 批量档案维护
- 大量录入型操作
- 维护型统计看板

### 2.2 护士/慢病管理师

护士或慢病管理师关注的是“哪些人该随访、哪些任务快到期、哪些患者失访了、哪些需要补录信息”。

护士/慢病管理师核心模块：

- 随访任务队列
- 复诊提醒列表
- 失访患者列表
- 待补录患者列表
- 患者流程状态板

重点信息：

- 随访截止日期
- 任务优先级
- 当前风险等级
- 当前是否已进入预测流程
- 下一步动作建议

### 2.3 档案员

档案员关注的是“患者资料是否完整、结构化事件是否可用于模型、录入是否规范”。

档案员核心模块：

- 患者档案列表
- 档案详情页
- 事件录入页
- 数据完整性提示

重点信息：

- 患者基本信息
- 关键结构化事件
- 数据支持度
- 缺失的必要字段

## 5. 参考项目与借鉴点

### 5.1 OpenMRS

参考地址：

- https://github.com/openmrs/openmrs-esm-patient-chart
- https://guide.openmrs.org/collecting-data/the-patient-dashboard-in-depth/

适合借鉴的点：

- `patient banner + dashboard + workspace` 的结构
- 患者头部信息固定显示
- 诊疗工作区和信息浏览区分开

适配到本项目：

- 患者详情页顶部固定展示基本信息和风险摘要
- 中间展示四元组和预测结果
- 右侧展示建议和后续动作

### 5.2 OpenEMR Recall Board

参考地址：

- https://www.open-emr.org/wiki/index.php/Recall_Board

适合借鉴的点：

- 以任务列表形式管理待随访患者
- 强调优先级、到期时间和责任人

适配到本项目：

- 新增“随访任务”模块
- 把患者随访、复查、补录提醒独立出来

### 5.3 OpenEMR Patient Flow Board

参考地址：

- https://www.open-emr.org/wiki/index.php/Patient_Flow_Board

适合借鉴的点：

- 通过流程状态板展示患者当前所处处理阶段
- 适合多角色协同

适配到本项目：

- 新增“患者流程状态板”
- 按“待随访 / 待复核 / 待补录 / 已稳定”展示状态

### 5.4 Bahmni

参考地址：

- https://www.bahmni.org/
- https://github.com/bahmni

适合借鉴的点：

- 面向真实医疗业务的模块分层
- 不同工作角色看到不同入口

适配到本项目：

- 医生端、护士端、档案员端分模块设计
- 业务模块优先于展示型指标模块
